# Windows Integration Notes

Ignore aims to support normal developer workflows:

- Ctrl+C
- Ctrl+X
- Ctrl+V
- Explorer copy/paste
- Drag and drop
- Folder transfers
- Uploads

## What Works Now

- Native tray menu
- Start with Windows setting
- Global and project rule management
- Protected filtered copy engine
- Explorer clipboard file-list fallback
- Logs and metrics for supported operations

## Clipboard Fallback Behavior

When protection is enabled and Explorer places a file list on the clipboard, Ignore detects the clipboard sequence, creates a filtered staging copy under the user's cache directory, then replaces the clipboard with the staged paths. When the user pastes, Explorer receives the cleaned folders/files.

This makes normal copy and paste work for common Explorer workflows, but it is not identical to a native shell extension:

- Filtering happens after copy selection, before paste.
- Large folders must be staged once before paste.
- Cut/move semantics are treated as safe copy semantics by the fallback.
- Some applications use private clipboard formats that are not `CF_HDROP`.
- Browser uploads still need browser-specific integration.

## Why Full Explorer Interception Needs Native Extensions

Explorer copy/paste and drag-drop use shell data objects, COM interfaces, clipboard formats, and process-owned transfer pipelines. A background Go process cannot reliably rewrite these operations after the user presses Ctrl+V without risking corruption, duplicate transfers, permissions issues, or broken shell behavior.

Browser uploads have a separate limitation: file picker and drag-upload flows are controlled by the browser sandbox and web page. A desktop app cannot universally filter arbitrary browser uploads without a browser extension, native messaging host, or staging-folder workflow.

## Best Fallback Strategy

The current app provides the safe foundation:

- Detect and cache rules
- Keep the filtered copy engine available
- Keep metrics and logs
- Provide a safe `CF_HDROP` clipboard fallback for common Explorer copies
- Isolate integration packages for future native bridges

For fully transparent behavior, the next production step is a signed native extension.

## Shell Extension Roadmap

1. Add a COM-based Explorer copy hook handler for file operation preflight.
2. Add an `IDataObject` drag-drop wrapper to filter dropped file lists.
3. Add a context-menu verb for explicit "Copy with Ignore" only as an optional fallback, not as the primary workflow.
4. Add a native messaging host for browser extensions.
5. Add browser extensions for Chrome/Edge/Firefox upload staging.
6. Keep all filtering calls routed through the existing Go copy engine and ignore matcher.
7. Sign binaries and installer for Windows SmartScreen reputation.

## Upload Awareness Roadmap

Universal upload interception is not available to ordinary desktop apps. Practical options are:

- Browser extension filters selected folders before upload.
- Local staging folder generated by Ignore for upload workflows.
- IDE/editor plugin integrations for common project share actions.
- Cloud sync provider integration for providers that expose local hooks.
